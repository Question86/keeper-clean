## WHY THE SPEC FAILS AGAINST THE BASELINE

1. **Bootstrap timing conflict**
   - Baseline + OPS protocols define `_BOOTSTRAP.md` as a reset-time artifact (created during reset when preparing the next loop).
   - The proposed design creates `_BOOTSTRAP.md` during `finalize`, which blurs the “finalize vs reset” boundary and risks starting a new loop context before the archive is moved/locked.

2. **Gate semantics conflict**
   - `_LOOP_GATE.md` is a validator snapshot. Generating a deliberately `BLOCKED` gate as a control mechanism is conceptually different from “validator says blocked because violations exist”.
   - If you intentionally block, you must define the deterministic rule for unblocking (and who/what is allowed to do it), otherwise the system can deadlock.

3. **Single-archive-per-loop risk**
   - A CLI that can generate an archive without checking for an existing archive-in-progress (or any finalized state) can accidentally produce a second archive for the same loop unless it is very strict.

---

## MINIMUM CHANGES TO MAKE IT PASS

- Split into two explicit commands aligned with existing semantics:
  - `loopctl finalize`:
    - Require `current.json STATE.status == ACTIVE`.
    - Refuse to run if any `ARCHIV_*.md` exists in workspace root.
    - Create `ARCHIV_XXXX.md` **in root**.
    - Set `current.json STATE.status = FINALIZED`.
    - Regenerate `_LOOP_GATE.md` (normal validator output; may be PASS or BLOCKED based on lint/consistency).
    - Do **not** create `_BOOTSTRAP.md`.
  - `loopctl reset` (human-invoked):
    - Move `ARCHIV_XXXX.md` → `archive/ARCHIV_XXXX.md`.
    - Increment loop number and set `status = READY_FOR_RESET`.
    - Create `_BOOTSTRAP.md`.
    - Regenerate `_LOOP_GATE.md`.

- If you still want a “hard stop” after finalize, encode it as a **state** rule (`FINALIZED` requires reset before ACTIVE), not as an intentionally fake `BLOCKED` gate.
